# Skill: error-book — 错题本驱动开发

> 用结构化错题本积累工程教训，自动触发落盘，防止同样的错误重复出现。

## 触发条件

满足以下任一条件时自动触发落盘流程：
1. Code Review 发现 P0 级问题
2. 测试失败
3. 同一错误连续出现 2 次
4. 用户指出流程违规
5. 实现与设计不一致
6. 文档描述与实际实现不一致

## 核心理念

**错误不可怕，重复犯同样的错误才可怕。** 每次出错都是改进系统的机会，但前提是把教训持久化到文件里，而不是留在对话中。

## 错题本结构

### 目录组织

按任务类型分文件，每个文件聚焦一个领域：

```
error-books/
├── _index.md          # 速查索引（编号 → 一句话 → 所在文件）
├── flow-gate.md       # 门控/流程执行
├── flow-code-habit.md # 代码修改习惯
├── flow-design.md     # 设计阶段
├── git-worktree.md    # Git/分支/CR
├── build-packaging.md # 构建/打包
├── tool-rules.md      # 工具使用
├── interface-rules.md # 前后端接口
└── common-mistakes.md # 跨角色通用
```

### 条目格式

每条控制在 5 行以内：

```
### DEV-XX：[标题] [频率标记]

- ❌ 错误做法：[一句话]
- ✅ 正确做法：[一句话]
- 💡 根因：[一句话]
- 🛡️ 防御：[checklist 条目或自动检查]
```

频率标记：🔴×N（高频）/ 🟡×N（中频）/ 🟢（已修复）

### 索引格式

`_index.md` 只放表格，不放详细内容：

```markdown
| 编号 | 一句话 | 频率 | 文件 |
|------|--------|------|------|
| DEV-4 | 跳过流程直接编码 | 🔴×20 | flow-gate.md |
```

## 落盘流程（6 步）

### Step 1：归因分析

走决策树判断根因类型：

| 归因路径 | 含义 | 动作 |
|----------|------|------|
| A | checklist 有但跳过了 | 分析为什么跳过，强化触发条件 |
| B | checklist 有但没拦住 | 修改措辞，扩大触发范围 |
| C | checklist 没有这条 | 新增检查项 |
| D | 不属于 checklist 能防的 | 写入错题本作为经验 |

### Step 2：落盘门禁声明

必须输出以下模板，不输出就写错题本 = 违规：

```
--- 落盘门禁 ---
触发原因：[CR 发现 P0 / 测试失败 / 用户指出 / ...]
归因路径：[A / B / C / D]
落盘目标：[目标文件名]
草稿预览：[完整条目草稿]
checklist 修改：[需要改第 X 条 / 新增 / 不需要]
---
```

### Step 3：读目标文件记录规则

实际打开目标错题本文件，读顶部的记录规则。不能跳过。

### Step 4：写入条目

根据归因路径写入对应文件，同时更新 `_index.md`。

### Step 5：防复发四步确认

每条必须回答：
1. 修了什么
2. 怎么引入的
3. 归因路径（A/B/C/D）
4. checklist 是否需要更新

### Step 6：口头承诺清零

检查本次对话中所有口头说的改进是否已落盘到文件。口头说了没落盘 = 没做。

## 高频错误模式（实战案例）

| 模式 | 频率 | 防御 |
|------|------|------|
| 跳过流程直接编码 | 🔴×20 | 完整门禁链缺一不可 |
| Write 工具超长内容失败 | 🔴×5 | >50 行先骨架再分段 |
| "我觉得我懂"陷阱 | 🟡×3 | 两次失败后必须停下搜索根因 |
| 补丁螺旋 | 🟡×2 | 连续 2 次补丁未解决 → 换路径 |
| 修复引入新问题 | 🟡×2 | 每次修复前做影响面分析 |

## 适用场景

- AI 辅助编码项目的经验积累
- 团队工程纪律建设
- 个人编程习惯改进
- 任何需要"不重复犯错"的工程场景
