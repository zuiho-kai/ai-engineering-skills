# Skill: patch-spiral-breaker — 补丁螺旋熔断器

> 当连续修复失败时强制停下换路径，防止在错误链路上无限叠补丁。

## 触发条件

对同一功能/链路提交第 2 次 fix 时自动触发。

## 核心理念

**补丁螺旋是最隐蔽的时间黑洞。** 每次写"根治"但只修表面症状，每层补丁遮蔽上层问题，最终 18 轮 commit 也解决不了一个本该换路径的问题。

## 真实案例

Electron transparent 窗口 setPosition 问题：
- 18 轮 commit，每次都说"这次根治了"
- 根因：未先验证平台限制就动手
- 每层补丁遮蔽上层问题，越修越复杂
- 最终解法：放弃 transparent 窗口方案，换完全不同的实现路径

## 熔断机制

### 第 2 次 fix：输出修复门禁

```
--- 修复门禁 ---
功能/链路：[修的是什么]
已有 fix 次数：2
每次 fix 摘要：
  第 1 次：[改了什么 → 为什么没解决]
  第 2 次：[改了什么 → 为什么没解决]
平台验证：[已做最小复现 / 已搜 Issues / 不涉及平台 API]
门禁结论：[继续当前路径]
--------------------
```

### 第 3 次 fix：强制熔断

```
门禁结论：强制停下换路径
```

累计 ≥3 次时，**禁止继续当前链路**，必须：

1. 停下来，不写任何代码
2. 验证平台限制（搜 Issues、查文档、做最小复现）
3. 列出至少 2 条完全不同的替代路径
4. 用户选择后才能继续

## 配套防御规则

| 规则 | 内容 |
|------|------|
| 两次失败必须搜索 | 连续 2 次失败后停下搜索根因，不能继续猜 |
| 第三方库先验底层 | 采用前必须查看平台相关源码 |
| 新需求先横向对比 | 开发前搜 2~3 个同类开源项目方案 |
| 修复前影响面分析 | 每次修复前分析可能引入的新问题 |

## "我觉得我懂"陷阱

最危险的心态是"这次我真的懂了"。识别信号：

- 没有搜索就开始修
- 修复方案和上次只有微小差异
- 说"应该是"而不是"验证过是"
- 没有做最小复现就下结论

**防御**：两次失败后，必须用搜索/文档/最小复现来验证假设，不能靠"我觉得"。

## 适用场景

- AI 辅助调试（AI 容易陷入补丁螺旋）
- 复杂 bug 修复
- 平台兼容性问题
- 任何"改了好几次还没好"的场景
