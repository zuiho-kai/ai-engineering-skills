# Skill: auto-persist — 信息持久化纪律

> 对话是临时的，文件是持久的。所有重要决策、教训、承诺必须落盘到文件，口头说了没写 = 没做。

## 触发条件

以下场景自动触发持久化检查：
1. 做出技术决策时
2. 发现流程改进点时
3. 承诺"以后要这样做"时
4. 讨论出重要结论时
5. 对话即将结束时（口头承诺清零检查）

## 核心理念

**口头承诺 = 没做。** 对话窗口关掉，所有没写进文件的内容都会消失。AI 助手尤其容易犯这个错——在对话中说"以后注意"，但下次对话完全不记得。

## 三条铁律

### 铁律 1：对话是临时的，文件是持久的

任何重要讨论必须落盘到文件。判断标准：如果这个信息在下次对话中还有用，就必须写进文件。

```
❌ "好的，以后我会注意先搜索再动手"
✅ 写入错题本：COMMON-9 两次失败后必须停下搜索根因
```

### 铁律 2：单一信息源

多个文件记录同一信息 = 迟早不一致。

```
❌ 在 README、CLAUDE.md、docs/guide.md 三个地方写同样的配置说明
✅ 只在一个地方维护详细内容，其他地方放链接
```

### 铁律 3：索引文件只放索引

索引文件（_index.md、MAP.md）只放表格和链接，不放详细内容。

```
❌ 在 _index.md 里写完整的错误分析
✅ _index.md 只放：编号 | 一句话 | 频率 | 所在文件
```

## 落盘检查清单

每次对话结束前，检查以下内容是否已落盘：

- [ ] 技术决策（为什么选 A 不选 B）
- [ ] 新发现的错误模式
- [ ] 流程改进点
- [ ] 口头承诺的规则变更
- [ ] 架构变更的原因

## 常见违规模式

| 违规 | 后果 | 防御 |
|------|------|------|
| 对话中说"以后注意" | 下次对话完全忘记 | 发现当下直接写入文件 |
| 改了内容没更新索引 | 索引和实际不一致 | 新建/修改文件时同步更新索引 |
| 详细内容放在索引文件里 | 索引膨胀，难以维护 | 索引只放表格和链接 |
| 多处记录同一信息 | 迟早不一致 | 单一信息源 + 链接 |

## 实施建议

### 最小起步

1. 创建一个 `decisions.md` 记录技术决策
2. 创建一个 `error-books/` 目录记录错误教训
3. 每次对话结束前做口头承诺清零检查

### 逐步扩展

4. 按领域拆分错题本文件
5. 建立索引文件
6. 设置自动落盘触发条件

## 适用场景

- AI 辅助编码（AI 的"记忆"依赖文件持久化）
- 团队知识管理
- 个人工程笔记
- 任何需要跨会话保持上下文的场景
