# Skill: cr-loop — Code Review 闭环

> 用独立 agent 做 Code Review，P0/P1 必须归零才能提交，杜绝"自己审自己"和"审完不改"。

## 触发条件

代码写完后、commit 之前自动触发。

## 核心理念

**代码写完后的下一步永远是 CR，不是 commit。** Code Review 是出口条件，不是可选步骤。自己不能审自己。

## 闭环流程

```
代码写完
    ↓
输出完成门禁模板
    ↓
启动独立 CR agent（不能自己审自己）
    ↓
CR 产出 P0 / P1 / P2
    ↓
P0/P1 必须修复
P2 逐条标注"修复"或"不修+理由"
    ↓
修复后重新 CR（全量复审，不是只看修复 diff）
    ↓
循环直到 P0=0 P1=0
    ↓
输出归零声明
    ↓
commit → push → 开 PR
```

## 严重等级定义

| 等级 | 含义 | 处置 |
|------|------|------|
| P0 | 会导致崩溃、数据丢失、安全漏洞 | 必须修复 |
| P1 | 逻辑错误、边界遗漏、接口不一致 | 必须修复 |
| P2 | 代码风格、命名、注释、可读性 | 逐条标注修复或不修+理由 |

## 归零声明

每条 P0/P1 修复后必须输出四步声明：

```
1. 修复了什么：[具体描述]
2. 怎么引入的：[原因分析]
3. 归因路径：[A：checklist有但跳过 / B：有但没拦住 / C：新场景 / D：架构问题]
4. checklist 是否需要更新：[是/否，具体内容]
```

四步缺一不可。

## 硬规则

- 归零声明必须由独立 CR agent 确认 P0=0 P1=0 后才能发出
- 禁止自己判断"大概率没问题"跳过 CR
- 修复后的 CR 审查范围 = 全部改动文件的完整复审，不是只看修复 diff
- P2 的"不修"必须给出理由，不能直接忽略

## CR 闭环四步（PR review 场景）

处理 PR review comment 时的闭环：

1. 修复 + 推送代码
2. 回复 PR review comment
3. 出错时执行自动落盘
4. 输出"CR 闭环完成"标记

未完成全部步骤就总结收工 = 流程违规。

## 适用场景

- AI 辅助编码的质量保障
- 团队 Code Review 流程标准化
- 任何需要"写完必审、审完必改、改完必验"的场景

## 来源

提炼自 GreyWind 项目完成门禁体系和 CR 闭环四步，核心教训：COMMON-15（跳过 CR 直接提交）、COMMON-18（CR 修复后跳过重新 Review）。
