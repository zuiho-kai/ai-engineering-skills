# Skill: task-gate — 分层任务门禁

> AI 编码助手的流程纪律框架。通过 4 层门禁卡点，防止跳步、漏审、口头承诺等高频工程事故。

## 触发条件

当用户要求 AI 执行任何会产生代码变更的任务时自动触发。

## 核心理念

**没有门禁声明就动手 = 流程违规。** 每一层门禁都有明确的输出模板，口头过不算，必须落盘。

## 4 层门禁

### 第 1 层：任务入口门禁

接到任何会产生 git diff 的任务后，**先输出声明，再做任何分析/读代码/写代码**。

```
--- 任务入口门禁 ---
任务：[一句话描述]
任务类型：[文档更新 / 单文件小修 / Bug修复 / 多文件功能 / 新模块]
判断依据：[为什么是这个类型]
对应流程：[见流程表]
门禁结论：[可以开始 | 需要先完成 XXX]
--------------------
```

**流程表**：

| 任务类型 | 判断标准 | 流程 |
|----------|----------|------|
| 文档更新 | 只改 .md，不涉及代码 | 直接改 |
| 单文件小修 | ≤1 文件，逻辑明确 | 直接实施 → 自测 |
| Bug 修复 | 有报错/复现，≤3 文件 | 实施 → CR → P0/P1 归零 |
| 多文件功能 | 跨文件但不涉及新模块 | 方案确认 → 实施 → CR → P0/P1 归零 |
| 新模块 | 全新能力域 | 走完整模块开发流程 |

> 拿不准按"多文件功能"处理。

### 第 2 层：方案确认门禁

多文件功能进入实施前，**必须输出方案并落盘到文件**，用户确认后才能写代码。

```
--- 方案确认门禁 ---
功能：[一句话描述]
验收标准：[编号列出，怎样算做完]
技术方案：[关键设计决策、数据流、改动文件清单]
风险点：[技术风险、平台边界]
落盘位置：[docs/design/xxx.md]
用户确认：[未确认 → 禁止实施 | 已确认 → 可以实施]
--------------------
```

**硬规则**：
- 方案必须写入文件，对话中列出不算落盘
- 用户未确认前禁止写任何业务代码
- 单文件小修和 Bug 修复不需要走此门禁

### 第 3 层：修复门禁

对同一功能提交第 2 次 fix 前，**必须先输出声明**。

```
--- 修复门禁 ---
功能/链路：[修的是什么]
已有 fix 次数：[计数]
每次 fix 摘要：[第 N 次改了什么 → 为什么没解决]
门禁结论：[继续当前路径(≤2次) | 强制停下换路径(≥3次)]
--------------------
```

**累计 ≥3 次时，门禁结论必须是"强制停下换路径"，禁止继续当前链路。**

### 第 4 层：完成门禁

代码写完后，**先输出声明，再做 commit/push/PR**。

```
--- 完成门禁 ---
自验证：[已运行验证 / 不适用]
Code Review：[未开始 / 进行中 / 已完成，P0=X P1=X P2=X]
P0/P1 归零：[未开始 / 已归零]
P2 处置：[逐条：修复 / 不修+理由]
门禁结论：[全部通过 → 可 commit | 未通过 → 继续处理]
--------------------
```

**硬规则**：代码写完后的下一步永远是 CR，不是 commit。

## 通用原则

- **提问即交权**：同一轮消息禁止"提问 + 执行"并存，提问后只等回答
- **并行执行**：多个独立修改 → 并行发出；P0/P1 修复列表 → 逐条核销不漏项
- **不做口头承诺**：流程改进发现当下直接写入文件，口头说了没落盘 = 没做

## 适用场景

- AI 辅助编码（Claude Code、Cursor、Copilot 等）
- 团队代码审查流程
- 任何需要防止"先写再说"习惯的工程场景
